## ✨ 功能特点

- **实时统计**：自动捕获每次 LLM 请求的输入/输出 token 消耗（使用 API 返回的真实值，适合计费参考）。
    
- **会话级统计**：按会话（私聊/群聊）独立统计，支持持久化每日数据。
    
- **分类分析**：可将系统提示词、上下文对话、插件消息等分类统计，帮助分析消耗构成。
    
- **WebUI 可视化**：提供独立 Web 服务，展示总消耗趋势、会话排行，并支持在线编辑配置规则。
    
- **命令查询**：通过 `/tokenstats` 命令快速查看当前会话统计。
    
- **规则建议**：`/tokenstats suggest` 可根据最近对话自动推荐插件归属规则。
    

## 🚀 安装方法

1. 在 AstrBot 的插件目录 `data/plugins/` 下新建文件夹，命名为 `astrbot_plugin_token_stats`。
    
2. 将插件文件（`main.py`、`metadata.yaml`、`config.yaml`、`requirements.txt`、`web/` 目录）放入该文件夹。
    
3. 重启 AstrBot（或通过 WebUI 重载插件）。
    

安装成功后，插件会自动运行，无需额外设置。

## ⚙️ 基础使用

### 1. 自动统计

每次与机器人对话时，插件会自动记录本次请求的 token 消耗。你不需要做任何操作。

### 2. 查看统计结果

在聊天中发送 `/tokenstats`，机器人会回复当前会话最近一次请求的统计详情。例如：

text

📊 Token消耗统计
会话: 114514
━━━━━━━━━━━━━━━━━━
📥 输入: 1145 tokens
📤 输出: 114 tokens
━━━━━━━━━━━━━━━━━━
🧠 人格提示词: 1145 tokens
💬 上下文对话: 1145 tokens
🔌 示例插件: 1145 tokens
━━━━━━━━━━━━━━━━━━
💡 注：输入/输出为 API 返回的真实值（作为计费参考），分类统计为插件计算值（用于分析消耗构成），二者可能不一致。

- **人格提示词**：所有 `role: "system"` 的消息（包括安全提示、默认人格等）。
    
- **上下文对话**：用户消息和机器人回复中未被任何插件规则匹配的部分。
    
- **各插件**：如果配置了归属规则，会单独列出对应插件的 token 消耗。
    

## 🔧 如何统计插件的 token？

如果你想单独统计某个插件（例如知识库、记忆插件）的 token 消耗，需要告诉插件如何识别该插件添加的消息。这通过编辑 `config.yaml` 文件实现。

### 1. 编辑 `config.yaml`

`config.yaml` 位于插件目录内，初始内容为：

yaml

plugins:
  # 在这里添加你的插件规则

每条规则的含义是：**如果某条消息同时满足以下所有条件，就把这条消息的 token 算作这个插件的**。

#### 可选条件（可以选一个或多个）

|条件|含义|怎么填|
|---|---|---|
|`role`|消息是谁发的？|从日志中“角色=”后面抄，例如 `"function"`、`"assistant"`|
|`name_contains`|如果消息有“名称”字段，名称里含有某个词|从日志中“名称=”后面复制关键词，例如 `"core_memory"`|
|`content_startswith`|消息内容以什么开头|从日志中“内容预览”里抄前几个字，例如 `"根据回忆"`|
|`content_contains`|消息内容里包含什么词|如果有特殊标记，例如 `"[NOTE]"`，直接写进去|

**示例**：假设示例插件A添加的消息特征是 `角色=function`、`名称=core_memory_recall`，你可以这样写：

yaml

plugins:
  - name: "示例插件A"        # 自己取的名字，会显示在统计结果里
    match:
      role: "function"
      name_contains: "core_memory"

如果该插件还会直接发一条助手消息，内容以“逸一时误一世”开头，你可以再加一条同名规则（token 会累加）：

yaml

  - name: "示例插件A"        # 和上面同名
    match:
      role: "assistant"
      content_startswith: "逸一时误一世"

### 2. 如何获取插件消息的特征？

为了让插件准确地将某条消息的 token 归入对应的插件，你需要先**观察这条消息长什么样**，然后把它的特征填入 `config.yaml`。以下是两种方法：

#### 方法一：使用 `/tokenstats suggest` 命令（最简单）

1. **触发插件**：在聊天中发送一条能让该插件工作的消息（例如问记忆插件“你还记得什么？”）。
    
2. **等待回复**，确保插件已运行。
    
3. **输入命令**：在同一个会话中输入 `/tokenstats suggest`。
    
4. **查看机器人返回的建议**。它会分析当前会话中所有非 system 消息，列出常见的角色、名称关键词、内容开头，并给出示例规则。
    
5. **复制示例**，把 `name` 改成你想要的插件显示名，然后粘贴到 `config.yaml` 中（可添加多条规则）。
    

**优点**：不需要看控制台日志，直接通过聊天获得建议，对普通用户非常友好。

#### 方法二：查看控制台日志（更精确）

1. **确保插件已启用**（安装后默认启用）。
    
2. **触发插件**：发送一条会触发目标插件的消息。
    
3. **打开控制台**（AstrBot Launcher 的“日志”标签页，或命令行窗口）。
    
4. **搜索关键日志**：按下 `Ctrl+F`，输入 `[✨Token统计✨]`，找到插件输出的日志块，形如：
    
    text
    
    [✨Token统计✨] ===== 消息 Token 明细 =====
    [✨Token统计✨] 消息[0] 角色=system, 名称=None, token数=1513
    [✨Token统计✨]   内容预览: # Persona Instructions ...
    [✨Token统计✨] 消息[1] 角色=user, 名称=None, token数=331
    [✨Token统计✨]   内容预览: 哼哼哼啊啊啊啊 <system_reminder>...
    [✨Token统计✨] 消息[2] 角色=function, 名称=core_memory_recall, token数=123
    [✨Token统计✨]   内容预览: 压力马斯内...
    
1. **定位到目标插件添加的消息**：根据消息的“角色”、“名称”和“内容预览”判断。例如上面 `角色=function, 名称=core_memory_recall` 就是示例插件A添加的消息。
    
2. **记录特征**：
    
    - **角色**：`function`
        
    - **名称关键词**：`core_memory_recall` 中包含 `core_memory`
        
    - **内容开头**：阿里马斯内...` 的前几个字
        
    - **内容包含**：如果有特殊标记（如 `24岁`），也可以记下
        
3. **编写规则**：根据记录的特征，在 `config.yaml` 中添加对应的条件。
    

#### 如何判断哪条消息属于哪个插件？

- 看消息的**名称**（如果是 `function` 消息），通常名称里会包含插件或功能的关键词（如 `core_memory_`、`knowledge_`、`weather_`）。
    
- 看消息的**内容预览**，如果内容里提到了该插件的功能（例如“根据回忆”属于记忆插件，“根据知识库”属于知识库插件），也可以作为依据。
    
- 如果你仍然不确定，可以暂时先用 `/tokenstats suggest` 列出的最常见角色和名称关键词，作为初始规则，然后观察 `/tokenstats` 的结果，再逐步调整。
    

### 3. 修改后重启 AstrBot

保存 `config.yaml`，重启 AstrBot，再发一条消息触发插件，然后输入 `/tokenstats`，就能看到插件的统计结果了。

## 🌐 WebUI 使用

插件启动后会独立运行一个 Web 服务，默认地址为 `http://localhost:8765`。打开浏览器访问该地址即可使用 WebUI。

> ⚠️ 注意：WebUI 目前没有身份验证，请勿在公网直接暴露此端口。如需远程访问，请自行添加反向代理并增加认证。

### 📊 统计页面（默认打开）

统计页面用于查看整体的 token 消耗趋势和会话排行，包含以下几个模块：

#### 1. 概览卡片

- **总消耗**：所有会话的历史 token 总和（包括已迁移的每日数据）。
    
- **今日消耗**：当天所有会话的 token 总和（包括尚未迁移到历史的数据）。
    
- **总输入**：所有会话的 API 输入 token 总和（基于 API 返回的 `prompt_tokens`）。
    
- **总输出**：所有会话的 API 输出 token 总和（基于 API 返回的 `completion_tokens`）。
    

这些数据每小时自动更新，刷新页面也会重新获取。

#### 2. 最近7天趋势图

- 展示最近 7 天（包括今天）的**总消耗、输入、输出** token 趋势。
    
- 鼠标悬停可查看具体数值。
    
- 图表为折线图，直观反映消耗变化。
    

#### 3. 会话排行

- 列出所有有过 token 消耗的会话，按**总消耗**从高到低排序。
    
- 表格列包括：
    
    - **会话ID**：显示友好的名称（如 `野兽先辈(私聊):114514`），点击可查看该会话的每日详情。
        
    - **总消耗 (tokens)**：该会话所有日期的 token 总和（包括今日未迁移数据）。
        
    - **插件消耗 (tokens)**：该会话中所有归入插件的 token 总和（来自配置规则）。
        
    - **最近活跃**：最后一次产生 token 消耗的日期。
        
- **表头可点击排序**：点击“会话ID”、“总消耗”、“插件消耗”或“最近活跃”表头，可切换升序/降序排列，方便找出消耗最大的会话。
    
- **点击会话ID**：弹出模态框，显示该会话最近 7 天的每日 token 消耗及详细分类（如人格提示词、上下文、各插件），便于分析会话消耗变化。
    

### ⚙️ 配置管理页面

点击顶部导航栏的“配置管理”标签页，进入配置界面。这里提供两种配置方式：**图形化规则配置**（推荐）和**高级模式 YAML 编辑**。

#### 1. 图形化规则配置（推荐）

该区域展示了当前已配置的插件归属规则，每条规则包含一个插件名称和一组匹配条件。你可以**添加、编辑、删除**规则，完全无需接触 YAML 语法。

##### 添加规则

- 点击 **+ 添加规则** 按钮。
    
- 在弹出的对话框中依次填写：
    
    - **插件名称**：将在 `/tokenstats` 命令和统计结果中显示的名称（如“天使记忆插件”）。
        
    - **角色**（可选）：消息的角色，如 `user`、`assistant`、`system`、`function`。
        
    - **名称包含**（可选）：消息的 `name` 字段（常见于 `function` 消息）中包含的关键词。
        
    - **内容开头**（可选）：消息正文的前几个字符。
        
    - **内容包含**（可选）：消息正文中包含的关键词。
        
- 点击确定后，规则会出现在列表中。
    

##### 编辑规则

- 在规则卡片上点击 **编辑** 按钮。
    
- 修改弹窗中的任意条件，点击确定保存。
    

##### 删除规则

- 在规则卡片上点击 **删除** 按钮，确认后即可移除。
    

##### 保存规则

- 点击 **保存规则** 按钮，插件会将当前规则写入 `config.yaml`，并提示保存成功。
    
- 保存后**必须重启 AstrBot** 才能使新规则生效。插件会提示需要重启。
    

##### 注意事项

- 规则按列表顺序依次匹配，一旦匹配成功即停止后续匹配。建议将最特殊的规则放在列表前面。
    
- 多条规则可以使用相同的“插件名称”，它们的 token 会累加到同一个插件名下。
    
- 规则中的条件必须**同时满足**才会匹配。
    

#### 2. 高级模式：直接编辑 YAML

如果你熟悉 YAML 语法，或需要批量调整规则，可以使用高级模式。

- 点击 **🔧 高级模式（直接编辑 YAML）** 展开编辑区。
    
- 编辑框中会显示当前的 `config.yaml` 内容，你可以直接修改。
    
- 点击 **保存配置** 按钮，插件会验证 YAML 格式，若正确则保存到文件并提示成功。
    
- 同样，保存后需要重启 AstrBot 才能生效。
    

> 💡 图形化规则配置和高级模式是**同步**的：在图形界面修改并保存后，YAML 编辑区的内容会自动更新；反之亦然。

#### 3. 会话消息记录

在配置管理页面下方，有一个 **会话消息记录** 区域，用于查看各个会话最近一次 LLM 请求的**完整消息列表**，帮助你判断如何编写匹配规则。

- **选择会话**：下拉菜单列出所有有过消息记录的会话，选择后会自动加载该会话的消息列表。
    
- **刷新当前会话**：手动重新加载当前选中会话的消息（如果会话又产生了新消息）。
    
- **消息列表**：以表格形式展示每条消息的**序号**、**方向**（输入/输出/系统）、**角色**、**名称**、**完整内容**。内容会自动换行，支持长文本查看。
    

这个功能尤其有用：

- 你可以看到每条消息的**角色**、**名称**（如果有）、**完整内容**，直接复制这些特征到规则中。
    
- 比如，你想让某条 `function` 消息归入“记忆插件”，看到它的名称是 `core_memory_recall`，就可以在图形化规则中添加“名称包含: core_memory”。
    

> 💡 如果你不确定某条消息来自哪个插件，可以查看它的内容，根据内容推断（如“根据回忆”→记忆插件）。

## ❓ 常见问题

### 1. 输入/输出 token 与分类统计不一致？

- **输入/输出 token** 来自 API 返回的真实值，用于计费参考。
    
- **分类统计** 是基于输入消息的估算值，用于分析消耗构成。二者计算方式不同，数值有差异属正常。
    

### 2. 为什么 API 返回的输入 token 明显大于插件计算值？

可能的原因包括：

1. **其他插件在请求前注入了额外消息**：某些插件（如记忆插件、知识库插件）会在 `on_llm_request` 钩子之后，向 `ProviderRequest` 的消息列表中添加内容（如检索到的记忆、系统提示等）。这些新增的消息会被计入 API 的 `prompt_tokens`，但本插件在 `on_llm_request` 阶段统计时尚未包含它们，因此计算值会偏小。
    
2. **计算方式差异**：API 的 `prompt_tokens` 是服务商按实际编码计算的结果，而本插件使用 `tiktoken`（cl100k_base）进行估算。不同模型可能使用不同 tokenizer，导致两者存在细微差异。
    
3. **系统提示词的重复**：如果 `req.system_prompt` 独立存在，本插件会主动将其加入消息列表并统计。但如果其他插件也添加了系统消息，可能导致重复计数，但通常 API 返回的是最终结果。
    
4. **插件调用顺序**：本插件的 `on_llm_request` 优先级较低（默认），如果其他插件先执行并修改了消息列表，本插件统计时能捕获到；但如果其他插件在更晚的钩子（如 `on_llm_response`）之后才修改，则无法捕获。
    

**建议**：将本插件的优先级调高，或检查其他插件的文档，确认它们是否在请求前修改了消息。你也可以通过 WebUI 的“会话消息记录”查看本插件捕获到的消息列表，与实际发送的内容对比，判断是否遗漏。

### 3. 本插件能统计所有 LLM 请求吗？

不能。本插件通过 AstrBot 的 `on_llm_request` 和 `on_llm_response` 钩子工作，只能捕获通过 AstrBot 主流程（即用户消息触发的对话）发出的 LLM 请求。如果其他插件为了自身功能而直接调用 LLM API（绕过 AstrBot 的 Provider 层），则这些请求不会触发钩子，因此无法被本插件统计。这是当前 AstrBot 插件机制的限制。未来若 AstrBot 提供统一的 LLM 调用入口，本插件将支持全局统计。

### 4. 其他插件直接调用的请求会不会在 AstrBot 日志中显示？

通常不会。AstrBot 的日志主要记录通过其内部事件系统触发的请求。如果某个插件在内部自行调用 LLM API，这些调用完全独立于 AstrBot 的核心流程，因此不会触发相关钩子，AstrBot 的日志中也不会自动记录它们的请求内容、响应或 token 消耗。除非该插件自身编写了日志输出，否则这些内部调用的细节不会出现在 AstrBot 的日志中。

### 5. 为什么人格提示词统计了 1513 tokens，但我的人格提示词只有几十个字？

你可能在 AstrBot 中开启了“安全模式”或“默认系统提示词”，这些也会被算入人格提示词。插件会把所有 `role` 为 `system` 的消息都归为人格提示词。如果你只想统计自己写的人格，可以查看日志确认哪条 system 消息是安全模式的，然后考虑关闭它或手动调整。

### 6. 上下文对话统计的是哪些消息？

所有既不是 `system`（人格），也没有匹配到任何插件规则的消息，都会归为“上下文对话”。通常包括用户消息、普通助手回复，以及那些没有特征标记的插件输出。

### 7. `name_contains` 里的“名称”是什么？是插件的名字吗？

不是。`name` 是消息自带的字段，常见于 `role` 为 `function` 的消息，它表示调用了哪个函数。例如天使记忆插件调用 `core_memory_recall` 时，消息就会带有 `name: "core_memory_recall"`。你要从日志中看到这个“名称”，然后把其中关键词填到 `name_contains` 里。

### 8. 同一个插件有多种不同类型的消息，怎么让它们都归到这个插件里？

写多条规则，每条规则 `name` 填相同的名字，插件会自动累加所有匹配到的消息 token。

### 9. 为什么我配置了规则，插件统计还是显示 2 tokens？

2 是固定开销，表示没有消息匹配到你的规则。请检查：

- 规则中的条件是否与日志中的消息特征完全一致（注意大小写、空格）。
    
- 规则顺序是否正确（一旦消息匹配了前面的规则，就不会再检查后面的）。
    
- 是否确实触发了插件（有些插件需要特定指令才会产生消息）。
    

### 10. `content_startswith` 和 `content_contains` 有什么区别？

- `content_startswith`：消息内容必须**以**指定文字开头。
    
- `content_contains`：消息内容只要**包含**指定文字即可。
    

### 11. 匹配条件可以同时用多个吗？

可以。规则中列出的条件必须**同时满足**才算匹配。你可以根据需要自由组合。

### 12. 我不需要插件统计功能，可以不用 `config.yaml` 吗？

可以。删除 `config.yaml` 或清空 `plugins: []`，所有非 system 消息都会归为“上下文对话”。

### 13. 为什么在控制台看到的日志那么多？怎么只看插件的输出？

所有插件输出的日志都带有 `[✨Token统计✨]` 前缀，在控制台搜索这个就能快速定位。

### 14. 插件统计的 token 和官方计费 token 一致吗？

插件使用 `tiktoken` 库，与 OpenAI 模型计费方式一致。如果使用其他模型，可能会有微小差异，但整体趋势相同。

### 15. WebUI 无法访问？

- 检查 AstrBot 是否正常运行。
    
- 确保端口 `8765` 未被占用。
    
- 尝试访问 `http://127.0.0.1:8765` 而不是 `localhost`。
    

### 16. 命令冲突？

如果其他插件也注册了 `/tokenstats`，请在插件管理中禁用其一，或修改插件源码中的命令名。

### 17. 如何重置统计数据？

删除 `daily_stats.json` 文件即可（注意备份）。插件会在下次运行时重新生成。

## ⚠️ 注意事项

1. **规则顺序很重要**：插件按 `config.yaml` 中列表的顺序依次判断，一旦消息匹配了前面的插件，就不会再检查后面的。请把最特殊的规则放在前面。
    
2. **重启生效**：修改 `config.yaml` 后必须重启 AstrBot 或重载插件才能生效。
    
3. **不同会话统计独立**：插件按会话 ID 存储统计结果，不同机器人、不同私聊/群聊的统计互不干扰。
    

## 🗑️ 卸载插件

直接删除 `data/plugins/astrbot_plugin_token_stats/` 文件夹，重启 AstrBot 即可。
